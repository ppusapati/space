## 2. E-PLT-001: Identity & Authentication

> **Epic Goal:** Enable secure user authentication with multi-factor support

### US-PLT-001: User Login with Email/Password

**Epic:** E-PLT-001  
**Use Case:** UC-PLT-001  
**Priority:** Must Have  
**Story Points:** 5

**As a** registered user  
**I want to** log in with my email and password  
**So that** I can access the platform securely

**Acceptance Criteria:**

```gherkin
Scenario: Successful login
  Given I am a registered user with valid credentials
  When I submit my email and password
  Then I receive a JWT access token
  And I receive a refresh token
  And I am redirected to the dashboard

Scenario: Invalid credentials
  Given I am a user with invalid credentials
  When I submit incorrect email or password
  Then I see an error "Invalid email or password"
  And no tokens are issued
  And the attempt is logged

Scenario: Account locked
  Given my account is locked due to failed attempts
  When I try to login
  Then I see an error "Account locked. Please reset password."
  And I am given a password reset option
```

**Technical Notes:**
- Use Argon2id for password hashing (memory: 64MB, iterations: 3, parallelism: 4)
- JWT access token expires in 15 minutes
- Refresh token expires in 7 days, single-use with rotation
- Log all login attempts to audit trail

**Dependencies:** None (foundational)

### US-PLT-002: Rate Limiting for Login Attempts

**Epic:** E-PLT-001  
**Use Case:** UC-PLT-001  
**Priority:** Must Have  
**Story Points:** 3

**As a** system administrator  
**I want** login attempts to be rate-limited  
**So that** brute-force attacks are prevented

**Acceptance Criteria:**

```gherkin
Scenario: Rate limit by IP
  Given an IP address has made 10 login attempts in 1 minute
  When another login attempt is made from that IP
  Then the request is rejected with HTTP 429
  And a "Retry-After" header indicates wait time

Scenario: Rate limit by account
  Given an account has 5 failed login attempts
  When another failed attempt occurs
  Then the account is temporarily locked for 15 minutes
  And an email notification is sent to the user

Scenario: Successful login resets counter
  Given an account has 3 failed attempts
  When a successful login occurs
  Then the failed attempt counter resets to 0
```

**Technical Notes:**
- Use Redis sliding window for rate limiting
- IP limit: 10 requests/minute
- Account limit: 5 failures before 15-min lockout
- Progressive lockout: 15min → 1hr → 24hr

**Dependencies:** US-PLT-001

### US-PLT-003: Token Refresh

**Use Case:** UC-PLT-001  
**Priority:** Must Have  
**Story Points:** 3

**As an** authenticated user  
**I want to** refresh my access token  
**So that** I can maintain my session without re-entering credentials

**Acceptance Criteria:**

```gherkin
Scenario: Successful token refresh
  Given I have a valid refresh token
  When I request a new access token
  Then I receive a new JWT access token
  And I receive a new refresh token (rotation)

Scenario: Expired refresh token
  Given my refresh token has expired
  When I request a new access token
```

**Technical Notes:**

- Store refresh token hash in Redis with session metadata
- Detect token reuse as potential theft

### US-PLT-005: Enroll in TOTP MFA

**Epic:** E-PLT-001  
**Use Case:** UC-PLT-002  
**Priority:** Must Have  
**Story Points:** 5

**As a** user  
**I want to** enroll in two-factor authentication  
**So that** my account is more secure

**Acceptance Criteria:**

```gherkin
Scenario: Generate TOTP secret
  Given I am logged in and MFA is not enabled
  When I start MFA enrollment
  Then I see a QR code containing the TOTP secret
  And I see the secret key for manual entry
  And I see 10 backup codes

Scenario: Verify TOTP setup
  Given I have scanned the QR code into my authenticator app
  When I enter a valid 6-digit code

Scenario: Save backup codes
  Given MFA enrollment is complete
  When I download backup codes
```

**Technical Notes:**

- Generate 160-bit secret, encode as base32
- Store encrypted secret in database
- Generate 10 backup codes (8 characters each)
- Backup codes are hashed before storage
