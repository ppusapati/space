## 2. Epic Catalog

### 2.1 Platform Module (PLT)

| Epic ID | Epic Name | Use Cases | Description |
|---------|-----------|-----------|-------------|
| **E-PLT-001** | Identity & Authentication | UC-PLT-001 to UC-PLT-004 | User login, MFA, password management, session handling |
| **E-PLT-002** | Tenant Management | UC-PLT-005 to UC-PLT-007 | Multi-tenant provisioning, settings, quotas |
| **E-PLT-003** | User Management | UC-PLT-008 to UC-PLT-010 | User CRUD, roles, permissions |
| **E-PLT-004** | Audit & Compliance | UC-PLT-011 | Audit logging, hash chain integrity |
| **E-PLT-005** | Notifications | UC-PLT-012 | Email, SMS, in-app notifications |

<details>
<summary><b>E-PLT-001: Identity & Authentication</b></summary>

**Description:** Core authentication system supporting multiple methods (password, MFA, SSO) with secure session management.

**Use Cases:**
- UC-PLT-001: User Login
- UC-PLT-002: Multi-Factor Authentication
- UC-PLT-003: Password Reset
- UC-PLT-004: Session Management

**Key Features:**
- Argon2id password hashing
- TOTP/WebAuthn MFA support
- JWT + Refresh token rotation
- OIDC/SAML federation
- Brute-force protection

**Dependencies:** None (foundational)

**Acceptance Criteria:**
- Users can authenticate via email/password
- MFA enrollment and verification works
- Sessions expire and refresh correctly
- Failed login attempts are rate-limited
- Audit logs capture all auth events

</details>

<details>
<summary><b>E-PLT-002: Tenant Management</b></summary>

**Description:** Multi-tenant organization management with isolation, settings, and quota enforcement.

**Use Cases:**
- UC-PLT-005: Create Tenant
- UC-PLT-006: Configure Tenant Settings
- UC-PLT-007: Manage Tenant Quotas

**Key Features:**
- Tenant provisioning with unique slug
- Row-Level Security (RLS) isolation
- Per-tenant settings (MFA policy, SSO)
- Resource quota enforcement
- Subscription tier management

**Dependencies:** E-PLT-001 (Authentication)

**Acceptance Criteria:**
- Tenants are fully isolated (data, config)
- Quotas prevent over-provisioning
- Settings propagate to all tenant users
- Tenant admins can manage their org

</details>

<details>
<summary><b>E-PLT-003: User Management</b></summary>

**Description:** User lifecycle management including CRUD operations, role assignment, and permission enforcement.

**Use Cases:**
- UC-PLT-008: Create User
- UC-PLT-009: Manage User Roles
- UC-PLT-010: Deactivate User

**Key Features:**
- User invitation workflow
- Role-based access control (RBAC)
- Permission inheritance
- User profile management
- Bulk user operations

**Dependencies:** E-PLT-001, E-PLT-002

**Acceptance Criteria:**
- Users can be created/invited
- Roles grant appropriate permissions
- Deactivated users cannot access system
- User actions respect tenant boundaries

</details>

<details>
<summary><b>E-PLT-004: Audit & Compliance</b></summary>

**Description:** Immutable audit trail with cryptographic integrity for compliance requirements.

**Use Cases:**
- UC-PLT-011: View Audit Logs

**Key Features:**
- Hash-chain audit log integrity
- Searchable audit history
- Export for compliance
- Retention policy management
- SIEM integration

**Dependencies:** E-PLT-001, E-PLT-002, E-PLT-003

**Acceptance Criteria:**
- All security events are logged
- Audit logs cannot be tampered
- Logs are searchable by time, user, action
- Exports meet compliance formats

</details>

<details>
<summary><b>E-PLT-005: Notifications</b></summary>

**Description:** Multi-channel notification system for alerts, events, and user communications.

**Use Cases:**
- UC-PLT-012: Configure Notifications

**Key Features:**
- Email notifications (SES)
- SMS notifications (SNS)
- In-app notifications
- Notification preferences
- Template management

**Dependencies:** E-PLT-001, E-PLT-002

**Acceptance Criteria:**
- Users receive notifications on preferred channels
- Critical alerts always delivered
- Users can manage preferences
- Templates are customizable per tenant

</details>

### 2.2 Ground Station Module (GS)

| Epic ID | Epic Name | Use Cases | Description |
|---------|-----------|-----------|-------------|
| **E-GS-001** | Satellite Catalog | UC-GS-001 to UC-GS-003 | Satellite registration, TLE management |
| **E-GS-002** | Ground Station Management | UC-GS-004 to UC-GS-005 | Station configuration, capabilities |
| **E-GS-003** | Pass Scheduling | UC-GS-006 to UC-GS-008 | Pass prediction, scheduling, conflicts |
| **E-GS-004** | Pass Execution | UC-GS-009 to UC-GS-011 | Real-time pass operations |
| **E-GS-005** | Telemetry Operations | UC-GS-012 to UC-GS-014 | TM streaming, history, limits |
| **E-GS-006** | Command Operations | UC-GS-015 to UC-GS-018 | Command submission, approval, execution |

<details>
<summary><b>E-GS-001: Satellite Catalog</b></summary>

**Description:** Satellite registration and TLE (Two-Line Element) management for orbit prediction.

**Use Cases:**
- UC-GS-001: Register Satellite
- UC-GS-002: Update TLE
- UC-GS-003: View Satellite Status

**Key Features:**
- Satellite registration with NORAD ID
- Automatic TLE refresh from Space-Track
- TLE age monitoring and alerts
- Satellite health status display
- Frequency and modulation config

**Dependencies:** E-PLT-002 (Tenant), E-CMN-001 (Health)

**Acceptance Criteria:**
- Satellites can be registered with unique NORAD ID
- TLEs refresh automatically every 6 hours
- Stale TLE warnings (>24h) displayed
- Satellite status reflects latest telemetry

</details>

<details>
<summary><b>E-GS-002: Ground Station Management</b></summary>

**Description:** Ground station configuration including location, capabilities, and antenna management.

**Use Cases:**
- UC-GS-004: Configure Ground Station
- UC-GS-005: Monitor Ground Station Health

**Key Features:**
- Station location (lat/lon/alt)
- Antenna specifications
- Capability declarations
- Real-time health monitoring
- Maintenance scheduling

**Dependencies:** E-PLT-002 (Tenant)

**Acceptance Criteria:**
- Stations can be configured with accurate location
- Antenna capabilities affect pass predictions
- Health status shows real-time metrics
- Maintenance windows block scheduling

</details>

<details>
<summary><b>E-GS-003: Pass Scheduling</b></summary>

**Description:** Satellite pass prediction and scheduling with conflict detection.

**Use Cases:**
- UC-GS-006: Predict Passes
- UC-GS-007: Schedule Pass
- UC-GS-008: Resolve Conflicts

**Key Features:**
- SGP4/SDP4 orbit propagation
- 7-day pass prediction window
- Minimum elevation filtering
- Conflict detection (same antenna)
- Priority-based scheduling

**Dependencies:** E-GS-001, E-GS-002

**Acceptance Criteria:**
- Pass predictions match industry tools (±1 min)
- Conflicts are detected before confirmation
- Higher priority passes can preempt
- Schedule changes notify affected users

</details>

<details>
<summary><b>E-GS-004: Pass Execution</b></summary>

**Description:** Real-time pass execution including antenna control, tracking, and status monitoring.

**Use Cases:**
- UC-GS-009: Execute Pass
- UC-GS-010: Monitor Pass Progress
- UC-GS-011: Generate Pass Report

**Key Features:**
- Automated pre-pass preparation (T-30min)
- Real-time antenna tracking
- AOS/LOS detection
- Live status streaming
- Post-pass report generation

**Dependencies:** E-GS-003, E-GS-005, Integration with Antenna Controller

**Acceptance Criteria:**
- Antenna slews to predicted position before AOS
- Tracking maintains lock throughout pass
- Telemetry flows during contact window
- Pass report generated within 5 minutes of LOS

</details>

<details>
<summary><b>E-GS-005: Telemetry Operations</b></summary>

**Description:** Real-time telemetry streaming, storage, and limit monitoring.

**Use Cases:**
- UC-GS-012: Stream Telemetry
- UC-GS-013: Query Telemetry History
- UC-GS-014: Configure Limit Checks

**Key Features:**
- Real-time WebSocket streaming
- TimescaleDB time-series storage
- Configurable limit thresholds
- Automatic anomaly detection
- Historical trend analysis

**Dependencies:** E-GS-004, E-CMN-002 (Streaming)

**Acceptance Criteria:**
- Telemetry streams with <100ms latency
- Historical queries return in <1s for 24h window
- Limit violations trigger alerts within 3 samples
- Data retained per policy (7d raw, 90d 1min, 5y 1hr)

</details>

<details>
<summary><b>E-GS-006: Command Operations</b></summary>

**Description:** Spacecraft commanding with two-person authorization for critical commands.

**Use Cases:**
- UC-GS-015: Submit Command
- UC-GS-016: Approve Command (Two-Person)
- UC-GS-017: Execute Command
- UC-GS-018: View Command History

**Key Features:**
- Command catalog with parameter validation
- Hazard level classification (SAFE/CAUTION/CRITICAL)
- Two-person authorization workflow
- Command queue management
- Execution tracking and acknowledgment

**Dependencies:** E-GS-004, E-PLT-003 (Users/Roles)

**Acceptance Criteria:**
- SAFE commands auto-approved
- CRITICAL commands require different approver
- Commands sent only during valid pass windows
- Acknowledgments tracked and logged

</details>
