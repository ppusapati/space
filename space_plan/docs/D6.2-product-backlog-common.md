## 2. E-CMN-001: Health & Monitoring

> **Epic Goal:** Provide health checks, metrics, and system status monitoring

### US-CMN-001: Liveness Probe Endpoint

**Epic:** E-CMN-001  
**Use Case:** UC-CMN-001  
**Priority:** Must Have  
**Story Points:** 2

**As a** Kubernetes orchestrator  
**I want to** check if a service is alive  
**So that** I can restart unhealthy containers

**Acceptance Criteria:**

```gherkin
Scenario: Healthy service
  Given the service process is running
  When GET /healthz is called
  Then HTTP 200 OK is returned
  And response body is "ok"
  And response time is < 100ms

Scenario: Unhealthy service
  Given the service is in a failed state
  When GET /healthz is called
  Then HTTP 503 Service Unavailable is returned
  And Kubernetes will restart the container

Scenario: No authentication required
  Given the /healthz endpoint
  When called without authentication
  Then the request succeeds
  And no auth check is performed
```

**Technical Notes:**
- Endpoint: `GET /healthz`
- No authentication (must be accessible to k8s)
- Only checks if process is alive, not dependencies
- Must respond quickly (< 100ms)

**Dependencies:** None (foundational)

---

### US-CMN-002: Readiness Probe Endpoint

**Epic:** E-CMN-001  
**Use Case:** UC-CMN-001  
**Priority:** Must Have  
**Story Points:** 3

**As a** Kubernetes orchestrator  
**I want to** check if a service is ready to receive traffic  
**So that** I don't route requests to unready pods

**Acceptance Criteria:**

```gherkin
Scenario: Service ready
  Given the service is running
  And database connection is established
  And Redis is connected
  And NATS is connected
  When GET /readyz is called
  Then HTTP 200 OK is returned

Scenario: Database not ready
  Given the database connection has failed
  When GET /readyz is called
  Then HTTP 503 Service Unavailable is returned
  And response indicates "database: not ready"

Scenario: Partial readiness
  Given Redis is not connected
  But database and NATS are connected
  When GET /readyz is called
  Then HTTP 503 is returned
  And response indicates which dependency failed
```

**Technical Notes:**
- Endpoint: `GET /readyz`
- Check all critical dependencies: DB, Redis, NATS
- Return quickly with cached status (refresh every 5s)
- No authentication required

**Dependencies:** US-CMN-001

## 3. E-CMN-002: Data Export

> **Epic Goal:** Enable asynchronous export of large datasets

### US-CMN-007: Create Export Job

**Epic:** E-CMN-002  
**Use Case:** UC-CMN-002  
**Priority:** Must Have  
**Story Points:** 5

**As a** user  
**I want to** export large datasets asynchronously  
**So that** I can download data without timeout

**Acceptance Criteria:**

```gherkin
Scenario: Create export job
  Given I am authenticated
  When I request an export with:
    | Field       | Value                    |
    | resource    | telemetry                |
    | filters     | satellite_id, time_range |
    | format      | csv                      |
  Then an export job is created
  And I receive a job ID
  And the job status is "pending"

Scenario: Export formats
  Given I am creating an export
  Then I can choose from:
    | Format   | Extension | Use Case          |
    | CSV      | .csv      | Spreadsheets      |
    | JSON     | .json     | API integration   |
    | Parquet  | .parquet  | Big data/analytics|
    | GeoJSON  | .geojson  | GIS tools         |

Scenario: Quota check
  Given I have 5 pending exports
  When I try to create another
  Then I see "Maximum pending exports reached"
  And the request is rejected
```

**Technical Notes:**
- Store job metadata in database
- Queue job to NATS for background processing
- Limit pending exports per user (default: 5)
- Validate filters before queueing

**Dependencies:** E-PLT-001 (Auth), E-PLT-002 (Tenant)

### US-CMN-008: Process Export Job

**Epic:** E-CMN-002  
**Use Case:** UC-CMN-002  
**Priority:** Must Have  
**Story Points:** 5

**As the** system  
**I want to** process export jobs in the background  
**So that** large exports don't block API requests

**Acceptance Criteria:**

```gherkin
Scenario: Process export
  Given an export job is queued
  When the worker picks up the job
  Then data is queried in batches
  And results are streamed to S3
  And progress is updated periodically

Scenario: Large export
  Given an export will produce 1 million rows
  When the export processes
  Then it completes within 30 minutes
  And memory usage stays bounded (streaming)

Scenario: Export failure
  Given an export encounters an error
  When the error is transient (timeout)
  Then the job is retried (max 3 times)
  When the error is permanent (invalid query)
  Then the job is marked as "failed"
  And the user is notified
```

**Technical Notes:**
- Batch size: 10,000 rows
- Stream directly to S3 (multipart upload)
- Progress updates every 5 seconds
- Timeout: 30 minutes per job
- Retry with exponential backoff

**Dependencies:** US-CMN-007

## 4. E-CMN-003: Scheduling

```gherkin

    | Field       | Value             |
    | name        | tle_refresh       |
    | schedule    | 0 */6 * * *       |

  Given I am creating a job

Scenario: Preview next runs
  Given I have entered a cron expression

```

**Technical Notes:**
- Support standard 5-field cron expressions
- Support extended syntax: @hourly, @daily, @weekly
- Store in database with tenant scope
- System jobs are tenant_id = NULL

### US-CMN-015: Execute Scheduled Job

**Epic:** E-CMN-003  
**Use Case:** UC-CMN-003  
**Priority:** Must Have  
**Story Points:** 5

**As the** scheduler  
**I want to** execute jobs at their scheduled times  
**So that** automation works reliably

**Acceptance Criteria:**

```gherkin
Scenario: Job executes on schedule
  Given a job is scheduled for 6:00 AM
  When 6:00 AM arrives
  Then the job handler is invoked
  And execution is logged
  And next_run is calculated

Scenario: Job execution timeout
  Given a job has a 5-minute timeout
  When execution exceeds 5 minutes
  Then the job is terminated
  And status is marked as "timeout"
  And an alert is triggered

Scenario: Concurrent execution prevention
  Given a job is already running
  When the next scheduled time arrives
  Then a new instance is not started
  And this is logged as "skipped (already running)"

Scenario: Job failure
  Given a job execution fails
  When the error is caught
  Then the failure is logged with error details
  And the job will retry at next scheduled time
```

**Technical Notes:**
- Use distributed locking (Redis) to prevent concurrent runs
- Timeout configurable per job (default: 5 min)
- Log start, end, duration, outcome
- Leader election for scheduler process

**Dependencies:** US-CMN-013
